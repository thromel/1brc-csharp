# 1BRC C# Performance Notes

## Target

Get as close as practical to published .NET 1BRC state of the art while preserving the original challenge contract:

- arbitrary UTF-8 station names, 1-100 bytes
- up to 10,000 unique stations
- exact min/mean/max by integer tenths
- output sorted by ordinal station name
- IEEE 754 roundTowardPositive-style one-decimal mean rounding

## Reference Implementations Studied

- `noahfalk/1brc`: published around 1.381s for default data and 3.739s for the 10K variation on Hetzner CCX33. Main techniques: AVX2 delimiter search, SIMD temperature parsing, 8-row ILP, specialized dictionaries, AOT.
- `buybackoff/1brc`: published high-end managed .NET implementation with AOT/JIT scripts, fixed dictionary, vectorized separator search, bit-trick integer parsing.
- `nietras/1brc.cs`: detailed C# implementation with vectorized parsing, small/large station maps, NativeAOT settings, and extensive 1BRC fixture coverage.

## Host Constraints

Local machine:

- `Darwin arm64`
- `.NET SDK 10.0.301` with runtime `10.0.9` for the final NativeAOT publish
- `Vector128=True`
- `Vector256=False`
- `AdvSimd=True`
- `Avx2=False`

This means AVX2/Vector256-heavy SoTA code cannot be ported directly. The implemented path uses portable unsafe code plus `Vector128` delimiter search and NativeAOT.

## Implemented Optimizations

- NativeAOT publish path with `OptimizationPreference=Speed` and `IlcInstructionSet=native`.
- Runtime-ID detection and native-binary preference in `calculate_average_csharp.sh`.
- Memory-mapped file processing with line-boundary partitioning.
- Parallel shard aggregation.
- Three-cursor parsing inside each shard: each worker splits its assigned line-aligned range into thirds and alternates rows from all three cursors into the same table. This is a portable scalar ILP analogue of the multi-row parsing used by the fastest x64 SIMD implementations.
- Padded-row fast path for rows at least 112 bytes from a shard end. This follows the reference-parser pattern of using the 1BRC maximum station-name length to avoid per-row end checks in the hot path, then falling back for the unpadded shard tail.
- `Vector128` semicolon search, with scalar fallback near segment ends.
- Split-tail delimiter scan: the first 16-byte semicolon check remains inline for the common short-name case, while the uncommon long-name scan is moved behind a no-inline helper to reduce hot-path code size.
- Bit-trick integer-tenth temperature parser for safe 8-byte reads, scalar fallback at EOF.
- Cheap station key based on first 8 bytes, last 8 bytes, and length, with a one-multiply hash inspired by the reference implementations' first-bytes hashing strategy.
- Padded fast-path station-key creation uses a single masked 8-byte load for short names, avoiding the byte-at-a-time loop required by the bounded tail parser.
- Exact correctness guard for names longer than 16 bytes via full byte comparison.
- Per-shard NativeMemory-backed struct hash table with cached key fields, 64-byte aligned bucket storage, deterministic disposal, and 32768-entry fixed capacity. Resize support is intentionally omitted because the 1BRC contract caps unique stations at 10,000, well below the 50% load point of the table.
- Station-table hit checks rely on exact key fields plus byte comparison for long names; the stored hash is used for bucket selection/merge indexing but is not re-checked after probing lands on a populated entry.
- Hot station-table update helpers are explicitly marked for inlining. This follows the reference implementations, where map update/lookup routines are forced into the parser loop to avoid one call per row.
- Pointer-backed station names during parsing: each table entry stores a pointer/length slice into the memory-mapped input instead of copying station-name bytes into a managed `byte[]`.
- Pointer-keyed final merge: shard tables are merged into a final `StationTable` while the memory map is still pinned, then each unique station name is decoded once for sorting/output.
- `BRC_THREADS` override for local worker-count sweeps; default uses `Environment.ProcessorCount` except for a measured 10-core Apple Silicon large-file heuristic.

## Benchmarks

Generated data:

- canonical 413-station Java generator, 100M rows, 1.3GB
- Python 10K-ish generator, 10M rows, 152MB

Best local warm NativeAOT timings on Apple Silicon:

- canonical 100M: about `0.17-0.19s` after cold launch/page-cache effects, with occasional scheduler outliers above that
- 10K-ish 10M: about `0.039-0.043s` after cold launch/page-cache effects

Rough canonical extrapolation to 1B rows on this host: about `1.8-2.1s`, not directly comparable to published Hetzner/x64 numbers.

The pointer-backed station-name change reduced parse-time allocation pressure and preserved correctness, but did not create a decisive wall-clock improvement on the canonical 100M local benchmark. The two-cursor parser did improve the canonical 100M warm path from roughly `0.31-0.33s` to roughly `0.26-0.29s`.

A four-cursor variant was also tested. It passed the official suite, but canonical 100M warmed around `0.26-0.28s` and did not beat the cleaner two-cursor implementation, while adding more branch and range-split complexity. It was rejected.

Recent map and scheduling experiments:

- Raising the open-addressing resize threshold from 50% to 75% passed correctness, but on the 10K-ish 10M file it increased CPU time without a wall-clock gain. It was rejected.
- A `NativeMemory` station table with deterministic disposal passed correctness and timed similarly to the managed-table path: canonical 100M warmed around `0.26-0.29s`, and 10K-ish 10M around `0.05-0.07s`. The wall-clock evidence was too weak for the added unsafe lifetime complexity, so the managed table was kept.
- Worker-count sweep on the M2 Pro canonical 100M file was refreshed after the three-cursor and inlining wins. The broad pass showed canonical 100M improving from 5 through 10 workers, with 9/10/11 close: medians were `0.201474s`/`0.201046s`/`0.199572s`, while 12 workers regressed to `0.253547s` and 14 to `0.226351s`. A narrower confirmation pass did not reproduce 11 as best: canonical 9/10/11 medians were `0.211227s`/`0.199234s`/`0.213280s`. The high-cardinality 10M file preferred fewer workers in the broad pass, with 4/6/8/10/12 medians `0.052609s`/`0.046585s`/`0.048406s`/`0.050713s`/`0.054518s`, but a narrower 6/8/10 pass favored 8 by a small amount (`0.046974s`/`0.045840s`/`0.049542s`). Because the best setting was workload-sensitive and the canonical confirmation still favored 10, the default remained `Environment.ProcessorCount` at that point with `BRC_THREADS` available for local calibration.
- The padded-row fast path passed the official fixture suite. On canonical 100M, it was wall-clock tied with baseline at roughly `0.26-0.29s` warm, with CPU time slightly lower in the first A/B pass (`~1.96-1.99s user` versus `~2.02-2.06s`). On 10K-ish 10M it remained around `0.05-0.06s`. It was kept because it matches the reference parser structure and does not regress correctness or measured throughput.
- Hoisting the semicolon `Vector128` constant out of the fast row parser passed correctness but was benchmark-neutral on both canonical 100M and 10K-ish 10M, so it was reverted.
- Fast-path station-key creation for short names passed the official fixture suite and gave a real canonical improvement. On canonical 100M, baseline warmed around `0.27-0.28s` with about `1.98-2.02s user`; fast-key warmed around `0.21-0.23s` with about `1.64-1.69s user` plus one `0.29s` wall-clock outlier. On 10K-ish 10M, it stayed at `0.05-0.06s` and reduced user CPU slightly from roughly `0.25-0.27s` to `0.24-0.25s`.
- Replacing the heavier three-multiply finalizing hash with a one-multiply first/last hash passed the official fixture suite and improved both measured workloads. On canonical 100M, the heavy-hash baseline warmed around `0.22-0.23s` with about `1.59-1.65s user`; the cheaper hash warmed mostly around `0.20-0.22s` with about `1.56-1.60s user` and one `0.24s` outlier. On 10K-ish 10M, it stayed at `0.05s` wall-clock and was tied or slightly lower in user CPU, so the collision tradeoff was acceptable on the tested workloads.
- A smaller 2048-entry initial station table was tested because the canonical data has only about 413 unique stations. It passed correctness but was rejected: canonical user CPU rose from roughly `1.55-1.61s` to roughly `1.67-1.72s` in the first A/B pass, with wall time also slightly worse. The 4096-entry table remains the better default tradeoff.
- A final pointer-keyed merge replaced the previous `Dictionary<string, Aggregate>` merge. This matches the reference pattern of merging native/pointer keys first and formatting only after aggregation. It passed the official fixture suite, and NativeAOT output matched the previous baseline byte-for-byte on both generated benchmark files. The timing evidence is small but favorable enough to keep: a final canonical 100M randomized pass showed baseline/candidate wall median `0.234707s`/`0.232258s`, min `0.221339s`/`0.219183s`, and user mean `1.622454s`/`1.601948s`; the 10K-ish 10M pass showed wall median `0.054420s`/`0.049869s`, min `0.049408s`/`0.042952s`, with user CPU roughly tied. Earlier passes were noisier, so this should be treated as an incremental cleanup/CPU reduction rather than a decisive throughput breakthrough.
- A three-cursor parser was tested as the untried middle point between the kept two-cursor parser and the rejected four-cursor parser. It passed the official fixture suite, and NativeAOT output matched the two-cursor baseline byte-for-byte on both generated benchmark files. On canonical 100M, the first randomized pass showed baseline/candidate wall median `0.218911s`/`0.212371s`, mean `0.225335s`/`0.214373s`, and user mean `1.602549s`/`1.574873s`; the high-cardinality 10M pass was roughly tied but slightly favorable, with wall median `0.054347s`/`0.053345s` and user mean `0.229212s`/`0.224780s`. A second canonical-heavy pass still favored the candidate on median and user CPU (`0.222959s`/`0.216639s`, `1.596488s`/`1.560959s`), so the three-cursor parser was kept.
- Marking the hot `StationTable.AddOrUpdate`, `StationTable.AddOrMerge`, and `StationEntry.Add` helpers with `AggressiveInlining` was a large NativeAOT codegen win and aligns with the map-update structure in the C# reference implementations. It passed the official fixture suite, and output matched the previous baseline byte-for-byte on canonical 100M and 10K-ish 10M generated files. Against the three-cursor baseline, the first randomized pass improved canonical 100M wall median from `0.211817s` to `0.193487s`, mean from `0.212126s` to `0.193284s`, and user mean from `1.581194s` to `1.380342s`. The high-cardinality 10M pass improved wall median from `0.052730s` to `0.048857s` and user mean from `0.232085s` to `0.200225s`. A second canonical-heavy pass confirmed the result with wall median `0.218168s`/`0.195581s` and user mean `1.571307s`/`1.379981s`.
- Adding `AggressiveOptimization` to the outer `ParseRange` three-cursor routine was tested after the inlining win. It passed correctness but was rejected because wall time was neutral to slightly worse: canonical median `0.195450s`/`0.195200s`, mean `0.195520s`/`0.196944s`; high-cardinality median `0.049047s`/`0.049248s`. The final published binary keeps the inlining hints only.
- Avoiding the duplicate `last` word load for exactly 8-byte station names in `StationKey.Create`/`CreateFast` was tested. It passed the official fixture suite but regressed both workloads, likely due to less favorable NativeAOT codegen: canonical 100M wall median worsened from `0.193193s` to `0.202326s` and user mean from `1.377275s` to `1.470483s`; high-cardinality 10M wall median worsened from `0.049037s` to `0.051172s` and user mean from `0.203240s` to `0.215818s`. It was reverted.
- Replacing the hot `StationEntry.Add` branch updates with `Math.Min`/`Math.Max`, matching some reference-map style, was also tested. It passed correctness, but the timing evidence was neutral rather than a real improvement. First pass canonical 100M mean/user were slightly worse (`0.192703s`/`0.196268s`, `1.387454s`/`1.389983s`) despite a slightly better median; high-cardinality was effectively tied. A second canonical-heavy pass stayed tied on mean/user (`0.194389s`/`0.194467s`, `1.389616s`/`1.388999s`) with worse outliers, so the explicit branch version was restored.
- Replacing the output sort comparator from `StringComparer.Ordinal.Compare` to `string.CompareOrdinal` was tested as a high-cardinality formatting micro-optimization. It passed the official fixture suite and output comparisons, but timing did not clear the keep bar. First pass showed small wall-time wins but worse user CPU: canonical 100M median `0.239604s`/`0.236673s`, user `1.276576s`/`1.293100s`; high-cardinality 10M median `0.054691s`/`0.052862s`, user `0.187535s`/`0.189470s`. A second high-cardinality pass was effectively tied on median (`0.054284s`/`0.054341s`) and had a large candidate outlier, so the original comparer call was restored.
- A temporary NativeAOT timing build split runtime by phase. Canonical 100M was almost entirely parser-bound: total median `178.295ms`, with map setup `0.010ms`, range creation `0.024ms`, parse `177.921ms`, merge `0.131ms`, and format `0.173ms`. The high-cardinality 10M file was still parse-heavy but exposed merge/format costs: total median `42.596ms`, parse `38.544ms`, merge `1.802ms`, and format `2.365ms` for `8,887` output stations. This ruled out output sorting/formatting as the next canonical bottleneck.
- Packing `StationEntry` by narrowing `NameLength` to `byte` and `Min`/`Max` to `short` was tested in a candidate copy. Output matched baseline on canonical 100M and high-cardinality 10M, but timing was neutral to slightly worse: canonical median `0.189194s`/`0.191001s`; high-cardinality median `0.045840s`/`0.045951s`. It was rejected to avoid extra casts and layout churn without measurable gain.
- Fusing the first semicolon vector scan with short-name key creation was also tested. The idea was to reuse the already-loaded first 16 bytes for common short station names, but NativeAOT codegen and vector extraction made the hot path slower. Output matched baseline, but canonical 100M regressed from `0.190003s` to `0.208991s`, and high-cardinality 10M regressed from `0.043787s` to `0.044768s`. It was rejected.
- Splitting `FindSemicolonFast` into an inline first-vector check plus a no-inline tail scan was kept as a small parser code-size win. Output matched the reconstructed pre-split baseline on both generated benchmark files. The direct confirmation pass showed canonical 100M presplit/split medians `0.189359s`/`0.188297s` and trimmed means `0.190061s`/`0.189010s`; high-cardinality 10M medians `0.041734s`/`0.041602s` and trimmed means `0.042356s`/`0.042148s`. The gain is small, but consistent enough to keep because the change is localized and the uncommon long-name path remains covered by official complex/10K fixtures.
- Additional station-key hash mixers were profiled against the generated unique station sets before runtime A/B. The no-multiply `addfold` mixer had better canonical probe stats than the current hash (`1.0484` vs `1.0654` average probes for 413 stations) but worse 10K-ish probe stats (`1.2862` vs `1.1765` average probes for 8,869 stations). Runtime confirmed that tradeoff: focused A/B showed canonical 100M median/trimmed-mean improving from `0.188478s`/`0.187979s` to `0.180716s`/`0.181349s`, but high-cardinality 10M regressing from `0.040153s`/`0.040515s` to `0.042937s`/`0.043488s`. It was rejected to preserve all-round behavior.
- Two one-multiply hash variants were also tested. Removing the suffix rotate entirely (`mulxor`) matched output but did not beat baseline overall: first-pass canonical median `0.191347s`/`0.189542s`, high-cardinality median `0.039638s`/`0.040578s`. Changing the suffix rotate from `27` to `41` kept the same operation count and had near-baseline 10K-ish probe behavior while giving a small runtime win. The focused A/B showed canonical 100M baseline/rot41 medians `0.193256s`/`0.190883s` and trimmed means `0.199120s`/`0.193975s`; high-cardinality 10M medians `0.042406s`/`0.042571s` and trimmed means `0.043061s`/`0.042944s`. The hash kept the multiplier and moved to `RotateLeft(last, 41)` at that point, then was superseded by the later `rot42` test below.
- Replacing `Parallel.For` with fixed shard workers was tested to remove thread-pool partitioning overhead from the already line-aligned range schedule. The candidate starts one `Thread` per non-zero shard while the main thread parses shard 0, then joins before merging. Output matched the `Parallel.For` baseline on canonical 100M and high-cardinality 10M. First randomized A/B was slightly favorable: canonical median/trimmed-mean `0.200331s`/`0.203412s` vs `0.199853s`/`0.201142s`, high-cardinality `0.050637s`/`0.050457s` vs `0.050240s`/`0.050056s`. A focused confirmation strengthened the case, especially on the high-cardinality file: canonical `0.199899s`/`0.205146s` vs `0.198724s`/`0.202298s`, high-cardinality `0.051594s`/`0.051221s` vs `0.048310s`/`0.048501s`. The fixed-worker path was kept; single-range files now bypass parallel scheduling entirely.
- Worker-count calibration was repeated after fixed workers and the 32768-entry table. A broad current-binary sweep showed canonical 100M best around 10-11 workers and high-cardinality 10M best around 6-8 workers: canonical 7/8/9/10/11/12 medians were `0.199849s`/`0.191674s`/`0.180974s`/`0.179955s`/`0.178902s`/`0.183565s`, while high-cardinality 4/6/8/10/12 medians were `0.049697s`/`0.037808s`/`0.038076s`/`0.039232s`/`0.042153s`. Focused confirmation showed canonical 9/10/11/12 medians `0.183819s`/`0.175682s`/`0.172189s`/`0.180638s`, and high-cardinality 6/8/10 medians `0.038474s`/`0.038190s`/`0.040183s`. A length-aware default heuristic using 11 workers for large files and 8 for mid-size files matched output but did not beat the then-current default in direct no-env A/B: canonical median/trimmed-mean `0.173509s`/`0.173199s` vs `0.173063s`/`0.175933s`, high-cardinality `0.037487s`/`0.037619s` vs `0.037535s`/`0.037793s`. The default remained `Environment.ProcessorCount` at that point, with `BRC_THREADS` available when calibrating for a specific host/workload.
- Re-testing station-table initial capacity after the later hash and worker changes overturned the older 4096-entry default. Capacity candidates of 8192 and 16384 both matched output on canonical 100M and high-cardinality 10M. First randomized pass favored larger tables: canonical 4096/8192/16384 medians `0.192877s`/`0.186642s`/`0.186600s`, trimmed means `0.194492s`/`0.187445s`/`0.185877s`; high-cardinality medians `0.046174s`/`0.045061s`/`0.042486s`, trimmed means `0.046070s`/`0.045057s`/`0.042621s`. A focused 4096 vs 16384 confirmation still favored 16384: canonical medians `0.195315s`/`0.186973s` and trimmed means `0.195263s`/`0.188166s`; high-cardinality medians `0.044318s`/`0.042411s` and trimmed means `0.044100s`/`0.042428s`. The higher per-worker table memory is acceptable on this host and avoids resize/probe costs; 16384 was kept at that point, then superseded by the later 32768 test below.
- A follow-up capacity sweep tested whether 16384 still resized on the 10K-ish workload and whether larger tables hurt cache behavior. Candidates of 32768 and 65536 matched output on canonical 100M and high-cardinality 10M. First pass showed 32768 only narrowly better on canonical and worse on high-cardinality, while 65536 did not beat 16384 overall: canonical 16384/32768/65536 medians `0.179754s`/`0.177285s`/`0.180334s`, high-cardinality medians `0.043762s`/`0.044711s`/`0.044216s`. A focused 16384 vs 32768 confirmation favored 32768 on both workloads: canonical medians `0.182140s`/`0.179678s` and trimmed means `0.185228s`/`0.182582s`; high-cardinality medians `0.041135s`/`0.039089s` and trimmed means `0.041598s`/`0.039111s`. A short post-publish smoke for the final 32768 binary showed canonical 100M median `0.177544s` and high-cardinality 10M median `0.039158s`. The 32768-entry initial table is now the kept setting; 65536 was rejected.
- Tracking occupied table indexes was tested after increasing the table to 32768 entries, to avoid scanning sparse arrays during merge and final `ToArray()`. The candidate recorded each insertion index and iterated only `_count` slots. Output matched the full-scan baseline on canonical 100M and high-cardinality 10M, but the added insertion writes outweighed the avoided scans: canonical full-scan/used-index medians `0.175273s`/`0.176835s` and trimmed means `0.176050s`/`0.177005s`; high-cardinality medians `0.039004s`/`0.039695s` and trimmed means `0.039385s`/`0.039764s`. The simpler full-array scan was kept.
- A copied-buffer `RandomAccess` input path was prototyped after reviewing the noahfalk, Nietras, Buybackoff, and Xoofx implementations. The candidate line-aligned ranges with `RandomAccess.Read`, copied each shard into native memory, reused the existing parser, and kept buffers alive until after pointer-keyed merge/format. It passed the official fixture suite and matched mmap output byte-for-byte on canonical 100M and high-cardinality 10M. It was rejected because canonical 100M regressed badly: mmap/randomaccess medians `0.179054s`/`0.230868s` and trimmed means `0.179657s`/`0.230547s`. High-cardinality evidence was mixed rather than decisive: an earlier default-worker pass favored mmap (`0.045685s`/`0.049162s` median), an 8-worker pass favored mmap (`0.041753s`/`0.044889s`), while the final same-pass high-cardinality run slightly favored RandomAccess (`0.044888s`/`0.043562s`). The all-round result supports the Xoofx platform choice of mmap on macOS for this implementation.
- Setting spawned shard worker threads to `ThreadPriority.Highest`, as done in the Nietras implementation, was also tested as a scheduling/outlier reduction candidate. It passed official fixtures and matched generated outputs. The first randomized pass was favorable on median/trimmed mean, but confirmation was mixed: canonical base/priority medians `0.173447s`/`0.171005s` and trimmed means `0.173435s`/`0.171990s`, while high-cardinality medians `0.034684s`/`0.035400s` and trimmed means `0.035421s`/`0.035878s`. Because this host showed no robust all-workload win and thread priority is a global scheduling hint rather than parser work reduction, it was rejected.
- Replacing hot `StationKey` and `StationEntry` auto-properties with fields was tested to see whether NativeAOT was leaving accessor overhead in the table path. It passed official fixtures and matched generated outputs, but regressed both workloads: canonical base/field medians `0.175090s`/`0.184934s` and trimmed means `0.176524s`/`0.187506s`; high-cardinality medians `0.044168s`/`0.045265s` and trimmed means `0.043186s`/`0.043967s`. The property-based structs were kept.
- Removing the 24-byte `StationKey` return and passing hash/first/last directly into `StationTable.AddOrUpdate` was also tested. It passed correctness, but the out-parameter transport shape was neutral to worse: canonical base/key-out medians `0.173011s`/`0.173677s` and trimmed means `0.173560s`/`0.175137s`; high-cardinality medians `0.036656s`/`0.036500s` but trimmed means `0.036419s`/`0.036858s`. The compact return-struct version was kept.
- A scalar 8-byte semicolon precheck was added ahead of the `Vector128` delimiter scan in `FindSemicolonFast`. This uses the standard zero-byte bit trick on `word ^ 0x3B3B...` to avoid vector compare/movemask extraction when the station-name delimiter is in the first eight bytes, then scans bytes 8-23 with `Vector128` and uses a shifted no-inline tail for longer names. It passed the official fixture suite and matched generated outputs. First randomized A/B showed canonical 100M base/scalar8 medians `0.173866s`/`0.156619s` and trimmed means `0.174695s`/`0.156567s`; high-cardinality 10M medians `0.035919s`/`0.035399s` and trimmed means `0.035884s`/`0.035308s`. Confirmation held: canonical medians `0.173440s`/`0.158105s` and trimmed means `0.176719s`/`0.159392s`; high-cardinality medians `0.035665s`/`0.034930s` and trimmed means `0.035562s`/`0.034558s`. This delimiter fast path was kept initially, then superseded by the 24-byte scalar head below.
- Two follow-up delimiter heads tested whether replacing more of the first-vector scan with scalar word checks would beat scalar8. Scalar16, which checks bytes 0-15 with two word scans before the original vector tail, passed correctness but was not useful: first-pass canonical scalar8/scalar16 medians `0.155720s`/`0.157056s` and trimmed means `0.157060s`/`0.158807s`; high-cardinality medians `0.034516s`/`0.036149s` and trimmed means `0.034855s`/`0.036162s`. Scalar24, which checks bytes 0-23 with three word scans before the shifted vector tail, passed correctness and won. First-pass canonical scalar8/scalar24 medians `0.155720s`/`0.149235s` and trimmed means `0.157060s`/`0.150165s`; high-cardinality medians `0.034516s`/`0.034940s` but trimmed means `0.034855s`/`0.034425s`. Confirmation was clearly favorable: canonical medians `0.160626s`/`0.150564s` and trimmed means `0.162743s`/`0.150357s`; high-cardinality medians `0.034971s`/`0.034438s` and trimmed means `0.034993s`/`0.034371s`. The final delimiter fast path keeps the scalar24 head.
- Scalar32 was tested as the next delimiter-head extension by checking bytes 0-31 with four scalar word scans before the vector tail. It passed official fixtures and matched generated outputs, but was mixed and not an all-workload improvement. The first three-way pass favored scalar24 on canonical median/trimmed mean (`0.149626s`/`0.149711s` vs `0.151557s`/`0.151015s`) and high-cardinality median (`0.035905s` vs `0.036688s`), while scalar32 only narrowly improved high-cardinality trimmed mean (`0.036251s` vs `0.035959s`). - Replacing the portable multiply/rotate station-key hash with ARM64 hardware CRC32C was promoted. - A CRC32C fold-order variant was tested after the hardware CRC32C promotion. Canonical baseline/fold-length medians were `0.151425s`/`0.153664s`, paired wall-delta median `+0.002383s`, and user-delta median `+0.007231s`, with only `16/40` wall wins. Production keeps the promoted `length` seed followed by `first` and `last`.
- The non-Castagnoli ARM64 CRC32 instruction was tested as a same-cost hash alternative to CRC32C, keeping the promoted seed/order (`length`, then `first`, then `last`) and the same scalar fallback. It passed official fixtures and generated-output comparisons, but paired runtime did not improve. Canonical baseline/CRC32 medians were `0.153152s`/`0.156790s`, paired wall-delta median `+0.002409s`, with only `16/40` wall wins. Small user/sys movements were mixed and did not offset wall-time regression, so production keeps CRC32C.
- A smaller short-name/long-name probe split was tested after re-reading the Nietras and Xoofx separate-vector-key map designs. Unlike the earlier rejected split-table idea, this candidate kept the single 32K table and only moved the rare long-name `SequenceEqual` path out of the short-name probe loop by adding short and long `AddOrUpdate`/`AddOrMerge` helpers. It passed official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs. Runtime was workload-split rather than promotion-grade: canonical improved (`0.159743s`/`0.155882s` medians, paired wall-delta median `-0.001806s`, `18/32` wins), but high-cardinality regressed (`0.042514s`/`0.043034s` medians, paired wall-delta median `+0.000644s`, `13/32` wins) and user CPU worsened by `+0.003802s` median. Production keeps the unified probe loop because the branch/code-shape tradeoff is not stable across the 10K-station workload.
- Worker-count calibration was refreshed again after the CRC32C hash promotion. A broad current-binary `BRC_THREADS` sweep suggested canonical 100M might have shifted upward, with 15 workers best among 9-15 in that pass (`0.147735s` median) while high-cardinality favored 8 workers (`0.040928s` median) over the current 9. Direct explicit-thread pairing partly supported only the large-file side: 15 beat 13 on canonical wall time (`-0.002987s` paired median, `33/50` wins), but 8 did not beat 9 on high-cardinality median wall time (`+0.000134s`, `24/50` wins) despite lower CPU. A no-env candidate changing only the 10-core large-file default from 13 to 15 passed official fixtures and generated-output comparisons, but failed production-grade confirmation: canonical baseline/candidate medians were `0.152840s`/`0.154483s`, paired wall-delta median `+0.000181s`, only `24/50` wins, and user/sys CPU both worsened. The current 13-worker large-file and 9-worker mid-size defaults remain production.
- A CRC32C bucket-index bit-selection candidate changed the fixed 32768-entry table from using the low 15 hash bits (`hash & CapacityMask`) to using the high 15 bits of the same CRC32C station hash. This kept exact key identity and table capacity unchanged and only altered probe distribution. It passed all official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs, but paired timing rejected it. Canonical raw medians looked favorable (`0.167004s`/`0.161305s`), yet paired wall-delta median was worse at `+0.002775s` with only `14/32` wins; user/sys CPU moved slightly down but did not translate to wall time. High-cardinality was effectively neutral on median (`-0.000057s` paired wall) but had worse mean wall time from candidate outliers and no CPU win. Production keeps low-bit indexing for the CRC32C hash.
- Removing the stored `Hash` field from `StationEntry` was re-tested under the current CRC32C production path, recomputing the merge hash from `NameLength`, `First`, and `Last` instead. The candidate passed all official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs. Runtime was not promotion-grade: canonical was mildly favorable on wall median and user CPU (`0.153264s`/`0.151094s` wall medians, paired wall-delta median `-0.000556s`, `19/32` wins, user-delta median `-0.007835s`), but high-cardinality regressed (`0.040772s`/`0.040401s` raw wall medians, paired wall-delta median `+0.000253s`, only `13/32` wins, user-delta median `+0.000147s`, sys-delta median `+0.000468s`). - Three short-name mask variants were tested in `StationKey.CreateFast()` to see whether the current `(1UL << (length * 8)) - 1` expression was costing NativeAOT codegen on the common `< 8`-byte station-name path. All variants passed official fixtures and generated-output comparisons. A `ReadOnlySpan<ulong>` mask table was mixed: canonical improved slightly (`0.156960s`/`0.155447s` medians, paired wall-delta median `-0.001778s`, `18/32` wins), but high-cardinality regressed on paired wall time (`0.041118s`/`0.041951s`, paired wall-delta median `+0.001650s`, `14/32` wins). A constant `switch` mask was a clear regression, with canonical paired wall-delta median `+0.033770s` and `0/32` wins, and high-cardinality paired wall-delta median `+0.005658s` with `2/32` wins. A right-shift mask (`ulong.MaxValue >> ((8 - length) * 8)`) was initially weakly favorable on wall time, but 50-pair confirmation rejected it: canonical paired wall-delta median `+0.000961s` with `22/50` wins, high-cardinality paired wall-delta median `+0.000517s` with `18/50` wins, and user/sys CPU did not improve. Production keeps the original left-shift/subtract mask.
- Replacing the private `record struct Range` with a plain readonly struct was tested as a NativeAOT code-surface cleanup because the range helper does not need synthesized record equality, deconstruction, or formatting members. The candidate passed all official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs, but runtime did not justify the change. Canonical wall timing regressed despite lower user/sys counters in a noisy pass: baseline/plain-range medians were `0.148994s`/`0.155498s`, paired wall-delta median `+0.003135s`, and only `14/32` wins; user-delta median was `-0.005874s`. High-cardinality was neutral-to-worse (`0.041278s`/`0.040621s` raw medians, paired wall-delta median `+0.000086s`, `16/32` wins, user-delta median `+0.001884s`). Production keeps the compact `record struct Range` form.
- Adopting the first joined partial table as the final results table was tested to avoid one native table allocation/clear and one full merge scan. The global candidate passed fixtures and generated-output comparisons, but was workload-split rather than promotion-grade: canonical regressed (`0.152230s`/`0.154630s` medians, paired wall-delta median `+0.001771s`, `14/32` wins), while high-cardinality improved strongly in that pass (`0.040297s`/`0.038544s`, paired wall-delta median `-0.001801s`, `26/32` wins). A gated variant kept the existing merge path for `13`-worker large-file runs and used adopt-first only for smaller multi-worker runs. It also passed fixtures and output parity, but did not reproduce the high-cardinality win: canonical was mixed (`0.149953s`/`0.150312s` raw medians, paired wall-delta median `-0.001659s`, but user/sys medians worsened), and high-cardinality regressed (`0.039094s`/`0.039583s`, paired wall-delta median `+0.000818s`, `12/32` wins). Production keeps the simpler always-new final table because the allocation/merge saving is not stable enough across workloads.
- Pointer-iterating the native table scans in `MergeInto()` and `ToArray()` was tested as a codegen check for the non-row-hot full-capacity scans. The candidate replaced indexed bucket loops with pointer increments, leaving insertion, probing, and formatting semantics unchanged. It passed all official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs. The first 32-pair pass looked promising on wall time, with canonical paired wall-delta median `-0.004650s` and high-cardinality `-0.000554s`, but confirmation rejected promotion: canonical flipped to a small regression (`0.158051s`/`0.157109s` raw medians, paired wall-delta median `+0.000528s`, `23/50` wins) and high-cardinality was effectively neutral (`0.041346s`/`0.041183s` raw medians, paired wall-delta median `-0.000073s`, `25/50` wins). - A two-lane ARM64 `AdvSimd` temperature parser was tested as the first concrete probe of the documented Vector128 multi-row frontier. The candidate used per-lane `AdvSimd.ShiftLogical`/`ShiftArithmetic` over two 64-bit temperature words inside the existing three-cursor fast loop, left safe-tail parsing unchanged, and kept the current station map. A scratch probe first verified the vector formula against the scalar parser for positive, negative, one-digit, and two-digit temperatures. The candidate then passed all official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs. Runtime was a decisive rejection: canonical baseline/vector-temp2 wall medians were `0.146837s`/`0.166222s`, paired wall-delta median `+0.019298s`, trimmed mean `+0.020460s`, and `0/32` wins, with user CPU up by `+0.152128s` median. High-cardinality also regressed (`0.040833s`/`0.041456s`, paired wall-delta median `+0.001128s`, `13/32` wins, user-delta median `+0.008496s`). The scalar word parser remains production; on this NativeAOT ARM64 path, per-lane shift setup, element extraction, and the required batch scheduling cost more than the saved scalar arithmetic.
- Current public C# reference implementations were calibrated on this ARM64/.NET 10 host to bound the value of whole-solution adoption. Shallow clones were checked at `noahfalk/1brc@b1d1617`, `xoofx/Fast1BRC@069d458`, and `nietras/1brc.cs@00d67d9`, then temporarily retargeted to `net10.0/osx-arm64` for NativeAOT. Noah's implementation published but crashed on the first official fixture because it executes x86 `Avx2` code on ARM64. Xoofx's implementation published but failed the official `measurements-1` fixture on this host (`0.0/19.8/19.8` instead of `19.8/19.8/19.8`), so it was not a runnable correctness-equivalent baseline. Nietras published after suppressing build-time style analyzers and removing a macOS-denied realtime priority request; it passed official fixtures, but its output rounding differed on generated workloads, so a tiny temporary Java-style mean-rounding shim was applied for timing comparability. With that shim, Nietras matched generated canonical 100M and high-cardinality 10M outputs. Its best observed thread settings were roughly 11 workers for canonical and 9 for high-cardinality, but paired timing was still much slower than this implementation: canonical wall medians `0.184136s`/`0.294547s`, paired wall-delta median `+0.100601s`, `1/20` Nietras wins, and user-delta median `+0.749659s`; high-cardinality medians `0.049091s`/`0.069135s`, paired wall-delta median `+0.020085s`, `0/20` wins, and user-delta median `+0.120141s`. Whole-reference adoption is therefore not useful on Apple Silicon; only individual algorithmic ideas that survive ARM64-specific correctness and paired timing should be considered.
- A full canonical 1B local baseline was finally run on the 10-core Apple M2 Pro after freeing enough disk. The official `CreateMeasurements` generator produced exactly `1,000,000,000` rows in `184.937s`; the file was about `13 GiB`. The current production NativeAOT binary passed all 12 official fixtures and matched its saved 1B output on later candidates. Five full-run `BRC_THREADS=13` measurements showed warm wall time mostly around `16.2s` with one outlier: `16.20s`, `16.23s`, `16.41s`, `16.22s`, `18.94s`. User time was about `13.6-15.3s`, while system time was high at `29.6-34.5s`, with roughly `842k` page faults per warm run. That makes the mmap/OS fault path the current full-1B bottleneck candidate rather than managed allocation or formatting.
- Two OS-boundary hints were tested against the full 1B file and rejected. Adding `madvise(MADV_SEQUENTIAL)` on the acquired mmap address passed all fixtures and matched 1B output. Its first randomized 6-vs-6 pass looked favorable on medians (`18.081s` baseline vs `17.444s` candidate), but a paired confirmation rejected it: baseline/candidate medians were `18.797s`/`19.674s`, paired delta median was `+0.858s`, and the candidate won only `1/4` pairs. Adding Darwin `fcntl(F_RDAHEAD, 1)` on the file descriptor before creating the mmap also passed correctness and 1B output parity, but was neutral rather than useful: baseline/candidate medians were `17.980s`/`18.074s`, paired delta median was `+0.002s`, and wins split `2/4`. Production keeps the plain .NET mmap path for now.
- A streaming `RandomAccess.Read` rewrite was prototyped behind an environment switch to attack the full-1B mmap page-fault cost without keeping a full copied file in memory. The candidate added owned station-name storage to `StationTable`, read each worker range in native chunks, parsed complete line-aligned chunks into owned-key partial tables, then merged into an owned-key final table. It passed all 12 official fixtures and matched mmap output on generated canonical 100M. It was rejected before a full 1B confirmation because the bounded timing did not justify the more complex path. With 16 MiB chunks, a 16-pair alternating 100M check had mmap/randomaccess medians `0.218461s`/`0.222301s`, paired delta median `+0.019857s`, and only `3/16` randomaccess wins; counters showed lower RSS but higher system time. Increasing chunks to 64 MiB made it worse: medians `0.188829s`/`0.241920s`, paired delta median `+0.057392s`, and `0/16` wins. The result suggests user-buffer copying and read-path overhead offset the avoided mmap faults at this scale; a future non-mmap attempt needs a more fundamental design than chunked `RandomAccess.Read` plus the existing parser.
- A direct native `mmap`/`munmap` path was tested to check whether the .NET `MemoryMappedFile` wrapper itself was a measurable bottleneck. The candidate mapped the existing file descriptor with libc and reused the current parser, scheduler, and pointer-key table. Both `MAP_PRIVATE` and `MAP_SHARED` variants passed all 12 official fixtures and matched generated canonical 100M output. `MAP_PRIVATE` was rejected on a 20-pair 100M pass: .NET/native medians were `0.262060s`/`0.273821s`, paired delta median `+0.018367s`, and native won `7/20`. `MAP_SHARED` was closer but still not promotion-grade: a 12-pair pass had medians `0.227828s`/`0.203202s`, but paired delta median was only `-0.001066s`, wins split `6/12`, and there was a large native first-run outlier. Warm `/usr/bin/time -lp` counters showed the same mmap/page-fault shape as the .NET path. Production keeps `MemoryMappedFile`; the wrapper is not the limiting library boundary.
- A short-name CRC32C hash variant was tested after the native mmap rejection. The candidate skipped the second CRC32C fold when `NameLength < 8`, since `First` and `Last` are identical on that path, while keeping exact key identity checks unchanged. It passed all 12 official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs. Runtime was workload-split and not promotion-grade: canonical baseline/candidate medians were `0.192747s`/`0.195914s`, paired wall-delta median `+0.005099s`, and only `14/32` candidate wins. High-cardinality improved in the same pass (`0.051563s`/`0.051075s`, paired wall-delta median `-0.002271s`, `21/32` wins), but the official-shaped canonical regression is too large for a global hash change. Production keeps the current double-fold CRC32C hash for all station lengths.
- A direct-mapped station lookup front cache was tested as a larger station-map specialization. The candidate added a per-worker 1024-entry native pointer cache from station hash to the last resolved table bucket, so repeated stations could update the cached bucket before running the normal 32K-table probe loop. It passed all 12 official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs, but runtime decisively rejected it. Canonical baseline/front-cache medians were `0.173299s`/`0.212336s`, paired wall-delta median `+0.042456s`, trimmed mean `+0.040687s`, and only `1/32` candidate wins. High-cardinality also regressed (`0.050976s`/`0.064344s`, paired wall-delta median `+0.013239s`, `6/32` wins). This confirms the previous probe-stat evidence: canonical already averages about one 32K-table probe, so an extra cache pointer load and equality check adds work rather than removing a bottleneck.
- A fallback-safe canonical-station direct aggregate path was prototyped after confirming that the high 16 bits of the current CRC32C hash are collision-free for the 413 official generator station names on this ARM64 host. The candidate generated a known-station lookup from `CreateMeasurements.java`, exact-matched known names by `(Hash high bits, NameLength, First, Last, full bytes for names over 16 bytes)`, aggregated matches into direct per-station slots, and kept the existing native table for unknown names. It passed all 12 official fixtures and matched generated canonical 100M plus high-cardinality 10M outputs, but runtime rejected it decisively. Canonical baseline/known-direct medians were `0.174776s`/`0.215039s`, paired wall-delta median `+0.044691s`, trimmed mean `+0.046251s`, and the candidate won `0/32` pairs. High-cardinality also regressed (`0.049202s`/`0.056007s`, paired wall-delta median `+0.005251s`, `4/32` wins). Even a perfect discriminator cannot beat the current one-probe native table when exact known-name checking and separate aggregate scans are added to every row.

## Remaining Gap To Published SoTA

The published fastest C# default-data implementation uses x64 AVX2/Vector256 and processes multiple rows per thread with SIMD and ILP. On this ARM64 host, the comparable next frontier would be:

- a dedicated AdvSIMD/Vector128 multi-row parser that processes 2-4 rows per lane group instead of one row at a time
- a more specialized compact station map with inline variable-length key storage and architecture-specific prefetch support, if C#/NativeAOT exposes a profitable ARM64 path
- a random-access read path benchmarked against mmap on macOS ARM64, since reference implementations disagree on which I/O path wins by platform and workload

Those are larger architecture rewrites, but the current implementation is a portable, validated ARM64-near-frontier baseline.
